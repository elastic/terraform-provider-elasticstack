# `generate-release-notes` — Workflow Requirements

Workflow implementation: `.github/workflows/generate-release-notes.yml`

## Schema

```yaml
on:
  release:
    types: [published]

permissions:
  contents: write
  pull-requests: read
```

## Requirements

- **[REQ-001] (Trigger)**: The workflow shall run when a GitHub Release is published (`release` event, `published` type).
- **[REQ-002] (Scope)**: The workflow shall generate notes for the published release’s `tag_name` from the event payload.
- **[REQ-003] (NotesGeneration)**: The workflow shall generate release notes using GitHub’s “generate release notes” API.
- **[REQ-004] (UpdateTarget)**: The workflow shall update the body of the same release instance that triggered the workflow (by `release.id`).
- **[REQ-005] (Idempotency)**: The workflow shall only update the release body when the body is empty or contains an explicit auto-generation marker (e.g. `<!-- autogenerated-release-notes -->`).
- **[REQ-006] (Safety)**: If the release body is non-empty and does not contain the marker, the workflow shall not overwrite it and shall exit successfully.
- **[REQ-007] (Marker)**: When updating the release body, the workflow shall include the marker to allow safe regeneration on subsequent runs.
- **[REQ-008] (Auth/Permissions)**: The workflow shall request only the minimal permissions required to generate notes and update the release (`contents: write`, `pull-requests: read`).
- **[REQ-009] (SupplyChain)**: The workflow shall pin third-party actions by commit SHA (not only a major version tag).
- **[REQ-010] (Observability)**: The workflow shall log whether it updated the release body or skipped due to a non-empty, non-marked body.
