# `elasticstack_kibana_security_list_item` — Schema and Functional Requirements

Resource implementation: `internal/kibana/securitylistitem`

## Purpose

Define the Terraform schema and runtime behavior for the `elasticstack_kibana_security_list_item` resource: managing individual entries in a Kibana value list (security list) via the Kibana Security Lists API. The resource supports creating, reading, updating, and deleting list items within a Kibana space, composite identity from space and item identifiers, import by composite `id`, optional item metadata, and accurate mapping between Terraform state and API responses including timestamp and audit fields.

## Schema

```hcl
resource "elasticstack_kibana_security_list_item" "example" {
  # Identity
  id          = <computed, string>            # composite: "<space_id>/<list_item_id>"
  list_item_id = <optional+computed, string>  # auto-generated by Kibana if not set; RequiresReplace
  space_id    = <optional+computed, string>   # default "default"; RequiresReplace

  # Reference to parent list
  list_id = <required, string> # RequiresReplace

  # Item value
  value = <required, string>

  # Optional
  meta = <optional, JSON (normalized) string>

  # Computed from API
  version_id = <computed, string>
  created_at = <computed, string>
  created_by = <computed, string>
  updated_at = <computed, string>
  updated_by = <computed, string>
}
```

Notes:

- The resource's Markdown description is embedded from `internal/kibana/securitylistitem/resource-description.md`.
- `meta` uses `jsontypes.NormalizedType` and is marshaled/unmarshaled as a JSON object.
- `id` is always computed and not configurable by the practitioner (no `Optional` flag); it is set from the composite of `space_id` and `list_item_id` returned by the API.

## Requirements

### Requirement: Kibana Security Lists Item API

The resource SHALL manage value list items through the Kibana Security Lists API: create list item, read list item, update list item, and delete list item. After a successful create or update API call, the resource SHALL re-fetch the list item using the read API and SHALL fail with an error diagnostic if the re-read returns an empty or nil response.

#### Scenario: Create then authoritative read

- GIVEN a successful create API response
- WHEN create completes
- THEN the provider SHALL re-fetch the list item with the read API and SHALL fail with an error if the read returns nil

#### Scenario: Update then authoritative read

- GIVEN a successful update API response
- WHEN update completes
- THEN the provider SHALL re-fetch the list item with the read API and SHALL fail with an error if the read returns nil

#### Scenario: Read removes missing items

- GIVEN a read/refresh
- WHEN the read API returns not found (nil response)
- THEN the provider SHALL remove the resource from state

#### Scenario: Delete idempotency

- GIVEN delete is called
- WHEN the API returns not found for the list item
- THEN the provider SHALL treat delete as successful (no error diagnostic)

### Requirement: API error surfacing

For create, read, and update, when the transport layer fails or the API returns an unexpected HTTP status code, the resource SHALL surface error diagnostics to Terraform. Delete SHALL surface errors except when the item is already absent (not found), which SHALL be treated as success.

#### Scenario: Non-success create/update/read

- GIVEN a non-success API response (other than read not-found handled by state removal)
- WHEN the operation completes
- THEN Terraform SHALL receive error diagnostics describing the failure including the HTTP status code

### Requirement: Provider configuration and Kibana client

On create, read, update, and delete, the resource SHALL obtain the Kibana OpenAPI client from the provider. If the client cannot be obtained, the resource SHALL return an error diagnostic and SHALL not proceed to the API.

#### Scenario: Unconfigured provider

- GIVEN the resource has no provider-supplied API client
- WHEN any CRUD operation runs
- THEN the operation SHALL fail with an error diagnostic

### Requirement: Identity and composite `id`

After reading from the API, the resource SHALL set `id` to the composite string `<space_id>/<list_item_id>`, where `space_id` is the Kibana space and `list_item_id` is the item identifier returned by the API. On update and delete, the resource SHALL parse the composite `id` from state using `CompositeIDFromStrFw` to derive `space_id` and `list_item_id`.

#### Scenario: State has composite id

- GIVEN a list item in state returned by the API
- WHEN state is written
- THEN `id` SHALL equal `<space_id>/<list_item_id>` for that item

#### Scenario: Update derives resource id from composite id

- GIVEN a plan with `id` = `"default/my-item-id"`
- WHEN the update request is built
- THEN the API update body SHALL use `id = "my-item-id"` (the `ResourceID` portion of the composite)

### Requirement: Import

The resource SHALL support Terraform import using an `id` value in the format `<space_id>/<list_item_id>` passed through to state via `ImportStatePassthroughID`. On refresh after import, the resource SHALL parse the composite `id` from state using `CompositeIDFromStrFw` to derive `space_id` and `list_item_id` for the read API call. If the composite `id` cannot be parsed, the resource SHALL surface the parse error as a diagnostic and SHALL not proceed to the API.

#### Scenario: Import then refresh

- GIVEN an import of id `my-space/my-item-id`
- WHEN the import and subsequent refresh complete
- THEN state SHALL hold `space_id = "my-space"`, `list_item_id = "my-item-id"`, and `id = "my-space/my-item-id"`

#### Scenario: Invalid composite id on read

- GIVEN a state with an `id` that cannot be parsed as `<space_id>/<list_item_id>`
- WHEN read runs
- THEN the provider SHALL surface an error diagnostic and SHALL not call the read API

### Requirement: Lifecycle — force replacement

Changing `list_item_id`, `space_id`, or `list_id` SHALL require destroying and recreating the resource rather than an in-place update.

#### Scenario: Replace on list_id change

- GIVEN an in-place plan change only to `list_id`
- WHEN Terraform evaluates the plan
- THEN the plan SHALL indicate replace (destroy/create) for the resource

#### Scenario: Replace on list_item_id change

- GIVEN an in-place plan change only to `list_item_id`
- WHEN Terraform evaluates the plan
- THEN the plan SHALL indicate replace (destroy/create) for the resource

### Requirement: Default space

When `space_id` is not set by the practitioner, the resource SHALL default `space_id` to `"default"` and SHALL include that value in all API calls and in the computed `id`.

#### Scenario: Default space_id

- GIVEN a resource configured without `space_id`
- WHEN the resource is created and state is read
- THEN `space_id` in state SHALL be `"default"`

### Requirement: Create — request body

When creating a list item, the resource SHALL send `list_id` and `value` in the request body. When `list_item_id` is known and set by the practitioner, the resource SHALL include it as the `id` field in the request body. When `meta` is known and set, the resource SHALL unmarshal and include it in the create request.

#### Scenario: Create with practitioner-supplied list_item_id

- GIVEN `list_item_id` is set to a known string in the plan
- WHEN the create request is built
- THEN the API request body SHALL include that value as `id`

#### Scenario: Create without optional fields

- GIVEN `list_item_id` and `meta` are null or unknown in the plan
- WHEN the create request is built
- THEN the API request body SHALL not include those optional fields

### Requirement: Update — request body and optimistic concurrency

The update request body SHALL include the `list_item_id` (derived from parsing the composite `id`) and the updated `value`. When `version_id` is known in state, the resource SHALL include it as `_version` in the update body to support optimistic concurrency. When `meta` is known in the plan, the resource SHALL include it in the update request.

#### Scenario: Update sends version_id for optimistic concurrency

- GIVEN `version_id` is known in state before an update
- WHEN the update request is built
- THEN the request body SHALL include `_version` set to that version_id value

#### Scenario: Update without meta

- GIVEN `meta` is null or unknown in the plan
- WHEN the update request is built
- THEN the request body SHALL not include `meta`

### Requirement: Read — composite ID parsing

On read, the resource SHALL parse the composite `id` from state using `CompositeIDFromStrFw`. The parsed `ClusterID` SHALL be used as `space_id` for the API call and SHALL update `space_id` in state. The parsed `ResourceID` SHALL be used as the list item `id` for the read API call.

#### Scenario: Read uses parsed space_id

- GIVEN a state with `id` = `"my-space/my-item"`
- WHEN read runs
- THEN the read API call SHALL use `space_id = "my-space"` and item `id = "my-item"`

### Requirement: Delete

The resource SHALL parse the composite `id` from state to derive `space_id` and `list_item_id`, then call the delete API. The resource SHALL surface any error diagnostic from the delete call except when the API returns not found, which SHALL be treated as success.

#### Scenario: Delete uses composite id

- GIVEN state with `id = "default/my-item-id"`
- WHEN delete runs
- THEN the delete API call SHALL use `space_id = "default"` and item `id = "my-item-id"`

### Requirement: Mapping — `meta` field

When the API response includes a non-nil `meta` object, the resource SHALL marshal it to a JSON string and store it as a normalized JSON value in state. When the API returns a nil `meta`, the resource SHALL store a null normalized JSON value in state. When the marshal operation fails, the resource SHALL surface an error diagnostic.

#### Scenario: Non-nil meta from API

- GIVEN the API returns a list item with a non-nil `meta` map
- WHEN the provider maps the response to state
- THEN `meta` in state SHALL be the JSON-serialized form of that map

#### Scenario: Nil meta from API

- GIVEN the API returns a list item with a nil `meta`
- WHEN the provider maps the response to state
- THEN `meta` in state SHALL be null

### Requirement: Mapping — timestamps and audit fields

The resource SHALL map `created_at`, `updated_at`, `created_by`, `updated_by`, and `version_id` from the API response to state. Timestamps SHALL be formatted using the fixed-layout `2006-01-02T15:04:05.000Z`.

#### Scenario: Timestamps stored in fixed-layout format

- GIVEN the API returns `created_at` and `updated_at` as time values
- WHEN the provider maps the response to state
- THEN `created_at` and `updated_at` in state SHALL be strings in the format `2006-01-02T15:04:05.000Z`
