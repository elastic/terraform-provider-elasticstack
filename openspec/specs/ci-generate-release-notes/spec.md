# `generate-release-notes` — Workflow Requirements

Workflow implementation: `.github/workflows/generate-release-notes.yml`

## Purpose

Define behavior for auto-generating and safely updating GitHub Release notes when a release is published.

## Schema

```yaml
on:
  release:
    types: [published]

permissions:
  contents: write
  pull-requests: read
```

## Requirements

### Requirement: Trigger and scope (REQ-001–REQ-002)

The workflow SHALL run when a GitHub Release is published (`release` event, `published` type). The workflow SHALL generate notes for the published release’s `tag_name` from the event payload.

#### Scenario: Release published

- GIVEN a release is published
- WHEN the workflow runs
- THEN it SHALL target the release’s `tag_name` from the payload

### Requirement: Notes generation and update target (REQ-003–REQ-004)

The workflow SHALL generate release notes using GitHub’s “generate release notes” API. The workflow SHALL update the body of the same release instance that triggered the workflow (by `release.id`).

#### Scenario: Update triggering release

- GIVEN the workflow runs for release id `R`
- WHEN notes are written back
- THEN release `R`’s body SHALL be updated

### Requirement: Idempotency and safety (REQ-005–REQ-006)

The workflow SHALL only update the release body when the body is empty or contains an explicit auto-generation marker (e.g. `<!-- autogenerated-release-notes -->`). If the release body is non-empty and does not contain the marker, the workflow SHALL not overwrite it and SHALL exit successfully.

#### Scenario: Skip non-empty body without marker

- GIVEN a non-empty release body without the marker
- WHEN the workflow runs
- THEN the body SHALL be unchanged and the job SHALL succeed

### Requirement: Marker and permissions (REQ-007–REQ-008)

When updating the release body, the workflow SHALL include the marker to allow safe regeneration on subsequent runs. The workflow SHALL request only the minimal permissions required to generate notes and update the release (`contents: write`, `pull-requests: read`).

#### Scenario: Marker written on update

- GIVEN the workflow updates the body
- WHEN the update completes
- THEN the marker SHALL be present for future idempotent runs

### Requirement: Supply chain and observability (REQ-009–REQ-010)

The workflow SHALL pin third-party actions by commit SHA (not only a major version tag). The workflow SHALL log whether it updated the release body or skipped due to a non-empty, non-marked body.

#### Scenario: Pinned actions

- GIVEN the workflow uses third-party actions
- WHEN the workflow file is reviewed
- THEN those actions SHALL reference commit SHAs
