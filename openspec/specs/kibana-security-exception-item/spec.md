# `elasticstack_kibana_security_exception_item` — Schema and Functional Requirements

Resource implementation: `internal/kibana/security_exception_item`

## Purpose

Define the Terraform schema and runtime behavior for the `elasticstack_kibana_security_exception_item` resource: manage Kibana exception list items via the Kibana Security Exceptions HTTP API, composite identity and import, provider-level Kibana connection only, version-gated `expire_time` support, entry-type validation at plan/validate time, and authoritative read-after-write to prevent plan drift.

## Schema

```hcl
resource "elasticstack_kibana_security_exception_item" "example" {
  # Identity
  id             = <computed, string>                   # composite: "<space_id>/<item_uuid>"; UseStateForUnknown
  space_id       = <optional, computed, string>         # default "default"; RequiresReplace
  item_id        = <optional, computed, string>         # human-readable identifier; RequiresReplace; UseStateForUnknown

  # Required fields
  list_id        = <required, string>                   # the parent exception list identifier; RequiresReplace
  name           = <required, string>
  description    = <required, string>
  type           = <required, string>                   # must be "simple"; RequiresReplace

  # Optional fields
  namespace_type = <optional, computed, string>         # "single" (default) or "agnostic"; RequiresReplace
  os_types       = <optional, set(string)>              # elements: "linux" | "macos" | "windows"
  tags           = <optional, set(string)>
  meta           = <optional, JSON (normalized) string>
  expire_time    = <optional, computed, RFC3339 string> # requires server >= 8.7.2

  entries {                                             # required list, minimum 1 element
    type     = <required, string>                       # "match" | "match_any" | "list" | "exists" | "nested" | "wildcard"
    field    = <required, string>
    operator = <optional, string>                       # "included" | "excluded"
    value    = <optional, string>                       # required when type is "match" or "wildcard"
    values   = <optional, list(string)>                 # required when type is "match_any"
    list {                                              # required when type is "list"
      id   = <required, string>
      type = <required, string>                         # "keyword" | "ip" | "ip_range"
    }
    entries {                                           # required when type is "nested"; min 1 element
      type     = <required, string>                     # "match" | "match_any" | "exists"
      field    = <required, string>
      operator = <required, string>                     # "included" | "excluded"
      value    = <optional, string>                     # required when nested type is "match"
      values   = <optional, list(string)>               # required when nested type is "match_any"
    }
  }

  comments {                                            # optional list of comments
    id      = <computed, string>                        # auto-generated by Kibana
    comment = <required, string>
  }

  # Computed (read-only from API)
  created_at     = <computed, string>
  created_by     = <computed, string>
  updated_at     = <computed, string>
  updated_by     = <computed, string>
  tie_breaker_id = <computed, string>
}
```

## Requirements

### Requirement: Kibana Security Exceptions API (REQ-001–REQ-004)

The resource SHALL manage exception items through the Kibana Security Exceptions HTTP API: create exception list item, read exception list item, update exception list item, and delete exception list item. Reference: [Kibana Exceptions API documentation](https://www.elastic.co/docs/api/doc/kibana/group/endpoint-security-exceptions-api).

#### Scenario: Create then authoritative read

- GIVEN a successful create API response
- WHEN create completes
- THEN the resource SHALL re-fetch the exception item with a read and SHALL remove itself from state if the item cannot be read back

#### Scenario: Update then authoritative read

- GIVEN a successful update API response
- WHEN update completes
- THEN the resource SHALL re-fetch the exception item with a read and SHALL remove itself from state if the item cannot be read back

#### Scenario: Read removes missing items

- GIVEN a read/refresh is triggered
- WHEN the API returns not found for the exception item
- THEN the resource SHALL remove itself from state

### Requirement: API error surfacing (REQ-005)

For create and update, when the API returns an empty successful body where item data is required, the resource SHALL surface an error diagnostic to Terraform. Delete SHALL surface all API errors.

#### Scenario: Empty create response

- GIVEN the create API returns a nil or empty body
- WHEN create runs
- THEN the resource SHALL surface an error diagnostic with summary "Failed to create exception item"

#### Scenario: Empty update response

- GIVEN the update API returns a nil or empty body
- WHEN update runs
- THEN the resource SHALL surface an error diagnostic with summary "Failed to update exception item"

### Requirement: Provider configuration and Kibana client (REQ-006)

On create, read, update, and delete, the resource SHALL use the provider's configured Kibana OAPI HTTP client. If that client cannot be obtained, the resource SHALL return an error diagnostic with summary "Failed to get Kibana client" and SHALL not proceed to the API.

#### Scenario: Unconfigured provider

- GIVEN the provider did not supply a usable Kibana API client
- WHEN any CRUD operation runs
- THEN the operation SHALL fail with an error diagnostic before making any API call

### Requirement: Identity and composite `id` (REQ-007–REQ-008)

After a successful read from the API, the resource SHALL set `id` to the composite string `<space_id>/<item_uuid>`, where `space_id` is the Kibana space and `item_uuid` is the exception item's UUID returned by the API. On read, the resource SHALL parse `space_id` from the composite `id` stored in state.

#### Scenario: State matches composite id

- GIVEN an exception item returned by the API
- WHEN state is written after create or update
- THEN `id` SHALL equal `<space_id>/<item_uuid>` for that item

### Requirement: Import (REQ-009)

The resource SHALL support Terraform import using an `id` in the format `<space_id>/<item_uuid>`. On import, `id` SHALL be passed through directly to state via `ImportStatePassthroughID` and the subsequent read SHALL populate all attributes.

#### Scenario: Valid import id

- GIVEN an import id of the form `my-space/<uuid>`
- WHEN import runs
- THEN state SHALL hold `id = "my-space/<uuid>"` and subsequent read SHALL populate all attributes from the API

#### Scenario: Agnostic item import fallback

- GIVEN an import where `namespace_type` is not yet known in state
- WHEN read is called after import and the first read (without namespace_type) returns not found
- THEN the resource SHALL retry the read with `namespace_type=agnostic` before removing from state

### Requirement: Lifecycle — force replacement (REQ-010)

Changing any of `space_id`, `item_id`, `list_id`, `type`, or `namespace_type` SHALL require destroying and recreating the resource rather than an in-place update.

#### Scenario: Replace on immutable field

- GIVEN a plan that changes only `list_id`
- WHEN Terraform evaluates the plan
- THEN the plan SHALL indicate replace (destroy/create) for the resource

### Requirement: Schema defaults (REQ-011)

When `space_id` is not provided in configuration, the resource SHALL default to `"default"`. When `namespace_type` is not provided, the resource SHALL default to `"single"`.

#### Scenario: Default space_id

- GIVEN no `space_id` in configuration
- WHEN the resource is planned
- THEN `space_id` SHALL be `"default"` in the plan

### Requirement: `type` validation (REQ-012)

The top-level `type` attribute SHALL only accept `"simple"`. Any other value SHALL be rejected at plan time with a validation diagnostic.

#### Scenario: Invalid type rejected

- GIVEN `type` set to a string other than `"simple"`
- WHEN Terraform validates the configuration
- THEN the provider SHALL return a validation diagnostic for `type`

### Requirement: `namespace_type` validation (REQ-013)

The `namespace_type` attribute SHALL only accept `"single"` or `"agnostic"`. Any other value SHALL be rejected at plan time with a validation diagnostic.

#### Scenario: Invalid namespace_type rejected

- GIVEN `namespace_type` set to a value other than `"single"` or `"agnostic"`
- WHEN Terraform validates the configuration
- THEN the provider SHALL return a validation diagnostic for `namespace_type`

### Requirement: `entries` minimum size (REQ-014)

The `entries` list attribute SHALL contain at least one entry. Configuring an empty `entries` list SHALL be rejected at plan time with a validation diagnostic.

#### Scenario: Empty entries rejected

- GIVEN `entries` configured as an empty list
- WHEN Terraform validates the configuration
- THEN the provider SHALL return a validation diagnostic stating at least one entry is required

### Requirement: Entry type validation (REQ-015–REQ-020)

The resource SHALL implement `ValidateConfig` and SHALL validate each `entries` element based on its `type` value. Validation SHALL only run on known values; unknown values (e.g. references to resources not yet created) SHALL be skipped. The following rules SHALL apply:

- For `type = "match"` or `type = "wildcard"`: `value` and `operator` MUST both be set (non-null); if either is null the provider SHALL return an error diagnostic with summary "Missing Required Field".
- For `type = "match_any"`: `values` and `operator` MUST both be set (non-null); if either is null the provider SHALL return an error diagnostic with summary "Missing Required Field".
- For `type = "list"`: `list` object and `operator` MUST both be set (non-null); if the `list` object is known, `list.id` and `list.type` MUST also be set; the provider SHALL return an error diagnostic for each missing required attribute.
- For `type = "exists"`: `operator` MUST be set (non-null).
- For `type = "nested"`: `entries` (the nested sub-list) MUST be set and non-null; the `operator` field SHALL be ignored and the provider SHALL emit a warning diagnostic with summary "Ignored Field"; nested sub-entries SHALL be validated recursively (only `match`, `match_any`, and `exists` are valid nested types; any other type SHALL produce an error diagnostic).

#### Scenario: match entry without value

- GIVEN an entry with `type = "match"` and `value` is null
- WHEN configuration is validated
- THEN the provider SHALL return a "Missing Required Field" error diagnostic referencing `value`

#### Scenario: nested entry with invalid sub-type

- GIVEN a nested entry (within a `type = "nested"` outer entry) with `type = "wildcard"`
- WHEN configuration is validated
- THEN the provider SHALL return an "Invalid Entry Type" error diagnostic

#### Scenario: nested entry operator warning

- GIVEN an entry with `type = "nested"` and `operator` set to a known value
- WHEN configuration is validated
- THEN the provider SHALL emit an "Ignored Field" warning diagnostic for `operator`

### Requirement: `operator` validation in schema (REQ-021)

The `operator` attribute on top-level entries SHALL only accept `"included"` or `"excluded"` (enforced by schema validator). The `operator` attribute on nested entries SHALL only accept `"included"` or `"excluded"` and SHALL be required (not optional) for nested entries.

#### Scenario: Invalid operator rejected

- GIVEN an entry with `operator` set to a string other than `"included"` or `"excluded"`
- WHEN Terraform validates the configuration
- THEN the provider SHALL return a validation diagnostic for `operator`

### Requirement: `entry.list.type` validation (REQ-022)

The `list.type` attribute within an entry of `type = "list"` SHALL only accept `"keyword"`, `"ip"`, or `"ip_range"`. Any other value SHALL be rejected at plan time.

#### Scenario: Invalid list type rejected

- GIVEN an entry of `type = "list"` with `list.type` set to a value other than `"keyword"`, `"ip"`, or `"ip_range"`
- WHEN Terraform validates the configuration
- THEN the provider SHALL return a validation diagnostic for `list.type`

### Requirement: Compatibility — `expire_time` (REQ-023)

When `expire_time` is set to a known non-null value, the resource SHALL verify the server version is at least **8.7.2** before create or update. If the server version is lower, the resource SHALL fail with an error diagnostic with summary `"expire_time is unsupported"` and detail naming the minimum required version.

#### Scenario: expire_time on unsupported stack

- GIVEN server version strictly below 8.7.2 and `expire_time` configured with a known value
- WHEN create or update runs
- THEN the provider SHALL return the expire_time unsupported error diagnostic

#### Scenario: expire_time on supported stack

- GIVEN server version >= 8.7.2 and `expire_time` configured
- WHEN create or update runs
- THEN `expire_time` SHALL be included in the request and stored in state as an RFC3339 timestamp

### Requirement: Mapping — optional fields on create and update (REQ-024)

When `item_id` is set and known, the resource SHALL include it in the create request body. When `namespace_type` is known, the resource SHALL include it in create and update request bodies. When `os_types` is known and non-empty, the resource SHALL include it in the request body. When `tags` is known and non-empty, the resource SHALL include it in the request body. When `meta` is known, the resource SHALL include it (parsed from JSON) in the request body. When `expire_time` is known, the resource SHALL include it formatted as `2006-01-02T15:04:05.000Z` in the request body after version verification.

#### Scenario: Omitting unset optional fields

- GIVEN `tags`, `meta`, and `expire_time` are not configured
- WHEN the create request is built
- THEN the request body SHALL omit those fields rather than sending null or empty values

### Requirement: Mapping — comments on create and update (REQ-025)

When `comments` is known and non-empty, the resource SHALL include comments in the create request body (using the create-comment API shape, which does not include `id`). On update, the resource SHALL include comments using the update-comment API shape (which also does not include `id`). Computed comment `id` values SHALL be populated from the API response on read.

#### Scenario: Comment id populated from API

- GIVEN comments configured without `id` (computed)
- WHEN the resource is read after create
- THEN each comment in state SHALL include the Kibana-assigned `id`

### Requirement: State mapping — `os_types` and `tags` (REQ-026)

When the API returns a non-empty `os_types` array, state SHALL store it as a non-null set. When the API returns an empty or absent `os_types`, state SHALL store a null set. The same behavior SHALL apply to `tags`.

#### Scenario: Empty tags from API

- GIVEN the API returns no tags for the item
- WHEN state is written
- THEN `tags` SHALL be null in state (not an empty set)

### Requirement: State mapping — `expire_time` (REQ-027)

When the API returns a non-null `expire_time`, the resource SHALL parse it as RFC3339 and store it in state as an RFC3339 time value. If parsing fails, the resource SHALL add an error diagnostic and store a null value. When the API returns a null or absent `expire_time`, state SHALL store a null RFC3339 value.

#### Scenario: expire_time null when API omits it

- GIVEN the API returns no `expire_time`
- WHEN state is written
- THEN `expire_time` SHALL be null in state

### Requirement: State mapping — `meta` (REQ-028)

When the API returns a non-null `meta` object, the resource SHALL marshal it to a JSON string and store it as a normalized JSON value in state. When the API returns a null `meta`, state SHALL store a normalized JSON null. If marshalling `meta` fails, the resource SHALL return an error diagnostic.

#### Scenario: Null meta from API

- GIVEN the API returns no `meta` field for the item
- WHEN state is written
- THEN `meta` SHALL be null in state

### Requirement: State mapping — `entries` from API (REQ-029)

When reading from Kibana, each entry in the API response SHALL be converted to the Terraform entry model preserving `type`, `field`, `operator`, `value`, `values`, `list`, and nested `entries` as appropriate. When the API returns no entries, the resource SHALL store a null list. For each entry type, the conversion SHALL set null values for fields not applicable to that type (e.g. `value = null` for `match_any` entries).

#### Scenario: Entries preserved on read

- GIVEN an exception item with a `match` entry in Kibana
- WHEN the provider reads the item
- THEN the stored `entries` SHALL reflect the `type`, `field`, `operator`, and `value` from the API response

### Requirement: State mapping — audit fields (REQ-030)

The resource SHALL store `created_at`, `created_by`, `updated_at`, and `updated_by` from the API response as strings in state, formatted as `2006-01-02T15:04:05.000Z`. It SHALL also store `tie_breaker_id` and `item_id` from the API response.

#### Scenario: Audit fields stored from API

- GIVEN an API response with all audit fields
- WHEN state is written
- THEN `created_at`, `created_by`, `updated_at`, `updated_by`, and `tie_breaker_id` SHALL reflect the API values

### Requirement: Delete uses composite id (REQ-031)

On delete, the resource SHALL parse the composite `id` from state to obtain `space_id` and `item_uuid`, then SHALL call the delete exception list item API using `item_uuid` as the item identifier scoped to `space_id`.

#### Scenario: Delete uses resource id from composite

- GIVEN state with `id = "my-space/<uuid>"`
- WHEN delete runs
- THEN the delete API call SHALL use `<uuid>` as the item id and be scoped to `my-space`
