## Requirements
### Requirement: Script modules live in a canonical directory

The system SHALL store all reusable GitHub Actions script modules under `.github/scripts/workflows/`.

#### Scenario: Module discovery
- **WHEN** an engineer needs to locate a shared workflow script
- **THEN** they SHALL find it under `.github/scripts/workflows/` or a subdirectory thereof

### Requirement: Script modules use CommonJS with standard signatures

The system SHALL author script modules as CommonJS files that export an async function accepting `{github, context, core}` and optionally other parameters keyed by environment variables.

#### Scenario: Module consumed from YAML
- **WHEN** a workflow step uses `actions/github-script` with `script: \|` containing `require('${{ github.workspace }}/.github/scripts/workflows/...')`
- **THEN** the required module SHALL execute correctly, reading `github`, `context`, and `core` from the injected scope

### Requirement: Shared logic is factored into lib modules

The system SHALL place pure, testable logic (functions without side effects) in `.github/scripts/workflows/lib/` and consume them via `require()` from orchestration modules.

#### Scenario: Unit testing shared logic
- **WHEN** a developer runs `node --test .github/scripts/workflows/lib/*.test.mjs`
- **THEN** the tests SHALL execute and assert the behavior of lib modules without mocking `actions/github-script` globals

### Requirement: Orchestration logic is separated from pure logic

The system SHALL keep event routing, GitHub API calls, and output-setting in orchestration modules (e.g. `.github/scripts/workflows/provider/classify-changes.js`), while keeping classification rules, gate formulas, and parsing logic in `lib/` modules.

#### Scenario: Updating a gate rule
- **WHEN** a maintainer changes the provider gate logic
- **THEN** they SHALL edit only the lib module (e.g. `lib/gate-provider.js`) and the orchestration module SHALL not contain hardcoded gate rules

### Requirement: Workflows without checkout gain a checkout step

The system SHALL ensure that any workflow job running `actions/github-script` with `require()` has a preceding `actions/checkout` step so that `${{ github.workspace }}` resolves to the repository root.

#### Scenario: Provider classify job runs
- **WHEN** the `classify` job in `provider.yml` executes
- **THEN** it SHALL checkout the repository before the first `actions/github-script` step

### Requirement: Generated header comments are removed

The system SHALL remove "This file is generated by..." header comments from `.github/workflows/*.yml` and `.github/workflows/*.md` because they are no longer generated by `compile-workflow-sources`.

#### Scenario: Reading provider.yml
- **WHEN** an engineer opens `.github/workflows/provider.yml`
- **THEN** they SHALL not see a generated-by header referencing `compile-workflow-sources`

### Requirement: No x-script-include or //include: directives remain

The system SHALL not contain `x-script-include:` or `//include:` anywhere in `.github/workflows/`, `.github/scripts/workflows/`, or Makefile targets.

#### Scenario: Search for legacy directives
- **WHEN** a maintainer runs `grep -r "x-script-include" .github/workflows/`
- **THEN** they SHALL find zero matches

### Requirement: Makefile targets are updated

The system SHALL remove all references to `compile-workflow-sources` from `Makefile` targets `workflow-generate`, `workflow-test`, and `check-workflows`. Test commands SHALL be updated to point to the new `.github/scripts/workflows/lib/` location.

#### Scenario: Running workflow tests
- **WHEN** a developer runs `make workflow-test`
- **THEN** it SHALL execute lib module tests and SHALL NOT reference `compile-workflow-sources`
