## Why

The `change-factory` workflow currently uses GitHub closing keywords (`Closes #N`) for issue ↔ PR linkage. That has two unintended consequences:

1. When the `change-factory` PR merges, GitHub auto-closes the originating issue. But a `change-factory` PR only delivers an OpenSpec proposal; the underlying user request is not yet implemented. The issue should remain open until the implementation PR (typically from `code-factory`) merges.
2. The `reproducer-factory` workflow (graduated as part of the `reproducer-factory-workflow` change) introduced a `'related-literal'` duplicate-linkage mode that matches `Related to #N`. Aligning `change-factory` with that convention removes the auto-close side-effect while keeping deterministic duplicate suppression.

## What Changes

- `change-factory` PR body MUST include `Related to #N` (not `Closes #N`).
- `change-factory` duplicate-PR suppression switches from `'github-keywords'` to `'related-literal'` mode (matches the literal phrase `Related to #N` only, mirroring `reproducer-factory`).
- The `change-factory` agent prompt no longer instructs the agent to write `Closes #N`.
- The graduated spec for `ci-change-factory-issue-intake` is updated so the duplicate-PR detection requirement and the linked-PR creation requirement reference `Related to #<issue-number>` rather than `Closes #<issue-number>`.

## Capabilities

### Modified Capabilities

- `ci-change-factory-issue-intake`: Duplicate-PR detection criterion and the agent's PR-body linkage guidance both shift from `Closes #N` to `Related to #N`, so a merging `change-factory` PR no longer auto-closes its source issue.

### New Capabilities

(none)

## Impact

- Source code: `.github/workflows-src/change-factory-issue/intake-constants.js` (one constant), `.github/workflows-src/change-factory-issue/workflow.md.tmpl` (two prompt mentions of `Closes #N`).
- Generated artifact: `.github/workflows/change-factory-issue.lock.yml` regenerated by `make workflow-generate`.
- Spec: `openspec/specs/ci-change-factory-issue-intake/spec.md` updated for the `Related to` linkage convention.
- Behavioral effect: existing open `change-factory` PRs that still carry `Closes #N` will no longer be detected as duplicates by the new check (they don't match `Related to`). This is acceptable because `change-factory` PRs are typically short-lived; a one-time stale state is not worth special-casing. New runs will produce `Related to` PRs going forward.
- No changes to `code-factory`, `research-factory`, or `reproducer-factory`. No changes to provider source code or runtime behavior.

## Dependencies

This change depends on the `'related-literal'` duplicate-linkage mode in `.github/workflows-src/lib/factory-issue-shared.js`, introduced by the `reproducer-factory-workflow` change (PR #2927). This branch is based on that branch and will rebase cleanly onto `main` once #2927 merges.
