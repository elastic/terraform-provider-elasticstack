## Why

The `elasticstack_fleet_integration_policy` resource exposes a top-level `enabled`
attribute that has never actually controlled the integration policy's enabled state.
The Kibana Fleet **package policy request** schema
(`PackagePolicyRequestMappedInputs`, used by both `POST` and `PUT
/api/fleet/package_policies`) has no `enabled` field — it only carries `name`,
`namespace`, `description`, `force`, `package`, `policy_id(s)`, `output_id`, `vars`,
and `inputs`. The `enabled` flag exists only on the response (`PackagePolicy`) and
is always returned as `true` for created policies. There is no API path to disable a
package policy at the top level; in Kibana the only way to stop a package policy
collecting data is to disable its individual inputs/streams or remove it from its
agent policy.

As a result, the provider's `enabled` attribute:

- Accepts a value from configuration but silently ignores it on `create` and
  `update` (`toAPIModel` never copies `model.Enabled` into the request body).
- Always reads back as `true` from the API on every refresh.
- Causes "Provider produced inconsistent result after apply" errors when a user
  configures `enabled = false` (the new acceptance test
  `TestAccResourceIntegrationPolicyEnabled` exposes this).

The attribute is a no-op footgun. Per-input/per-stream `enabled` toggles still work
and remain the correct way to disable telemetry from a policy.

## What Changes

- **Schema**: drop the top-level `enabled` attribute from
  `elasticstack_fleet_integration_policy`. Bump the resource schema from version 2
  to version 3.
- **State upgrade**: add a v2 → v3 state upgrader that drops `enabled` from prior
  state. The existing v0 → v2 and v1 → v2 chains are retargeted to produce the v3
  model (still dropping `enabled` along the way).
- **Model / mapping**: remove `Enabled` from `integrationPolicyModel` and from
  `populateFromAPI`. The kbapi response struct is unchanged; the provider simply no
  longer maps the response field into Terraform state.
- **Tests**:
  - Delete `TestAccResourceIntegrationPolicyEnabled` and its testdata directories
    (`create_disabled/`, `re_enable/`).
  - Drop top-level `enabled` assertions from `TestAccResourceIntegrationPolicy`
    (the per-input/per-stream `enabled` checks remain).
  - Update `schema_v1_test.go` to no longer assert `Enabled` carry-over and to use
    the renamed `toV3` conversion.
  - Add a v2 → v3 unit test that demonstrates `enabled` is dropped from upgraded
    state.
- **Docs/spec**: update `openspec/specs/fleet-integration-policy/spec.md` to remove
  `enabled` from the schema block, drop it from the state-mapping requirement, and
  add a state-upgrade requirement for v2 → v3.
- **CHANGELOG**: add a breaking-change entry under `## [Unreleased]`.

## Capabilities

### New Capabilities

*(none)*

### Modified Capabilities

- `fleet-integration-policy`:
  - Schema block updated (no `enabled` attribute).
  - REQ-022 (State mapping) updated to drop `enabled` from the mapped fields.
  - REQ-024 / REQ-025 (state upgrades from v0/v1 to current) updated to target
    schema version 3 and to no longer carry `enabled`.
  - New requirement: state upgrade v2 → v3 SHALL drop the `enabled` attribute from
    prior state.
