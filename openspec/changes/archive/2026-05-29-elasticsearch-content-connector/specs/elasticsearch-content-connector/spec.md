## ADDED Requirements

### Requirement: Resource and data source coverage of the connector envelope (REQ-001)

The provider SHALL expose a `elasticstack_elasticsearch_connector` resource that manages the full lifecycle of an Elasticsearch content connector by calling the [connector APIs](https://www.elastic.co/docs/api/doc/elasticsearch/group/endpoint-connector), and a companion `data.elasticstack_elasticsearch_connector` data source that reads the same connector by ID.

The resource SHALL use:
- `POST /_connector` (when `connector_id` is unset) or `PUT /_connector/{connector_id}` (when set) for Create.
- `GET /_connector/{connector_id}` for Read.
- `DELETE /_connector/{connector_id}` for Delete.
- Per-aspect partial-update endpoints for Update: `_name`, `_index_name`, `_service_type`, `_native`, `_pipeline`, `_scheduling`, `_features`, `_api_key_id`, `_configuration`.

The data source SHALL use `GET /_connector/{connector_id}` for Read.

Both SHALL require the `manage_connector` cluster privilege (or `monitor_connector` for read-only operations).

Non-success API responses other than 404 on Read SHALL be surfaced as Terraform diagnostics. The resource Read SHALL remove the resource from state on 404. Delete SHALL treat 404 as success (idempotent).

#### Scenario: Resource create uses POST when connector_id is unset

- **WHEN** the resource is created without `connector_id`
- **THEN** the provider SHALL call `POST /_connector` and capture the returned `id` into state

#### Scenario: Resource create uses PUT with explicit connector_id

- **WHEN** the resource is created with `connector_id = "music"`
- **THEN** the provider SHALL call `PUT /_connector/music`

#### Scenario: Read 404 removes from state

- **WHEN** Read is called for a connector that no longer exists
- **AND** the API returns 404
- **THEN** the resource SHALL be removed from state without raising a diagnostic error

#### Scenario: Delete 404 is success

- **WHEN** Delete is called for a connector that has already been removed
- **AND** the API returns 404
- **THEN** the operation SHALL succeed without raising a diagnostic error

#### Scenario: Data source returns connector details

- **GIVEN** a connector exists with `connector_id = "music"`
- **WHEN** the data source reads it
- **THEN** state SHALL include all read-time attributes per REQ-010

#### Scenario: Data source returns error for missing connector

- **GIVEN** no connector exists with the supplied `connector_id`
- **WHEN** the data source reads
- **THEN** diagnostics SHALL include an error indicating the connector was not found

### Requirement: Identity (REQ-002)

The resource and data source SHALL expose a computed `id` in the format `<cluster_uuid>/<connector_id>`. `id` SHALL be computed after a successful Create (resource) or Read (data source). `connector_id` SHALL be `Optional + Computed` on the resource â€” supplied by the user OR auto-generated by Elasticsearch and captured into state.

#### Scenario: ID set after create with explicit connector_id

- **GIVEN** the resource is created with `connector_id = "music"`
- **WHEN** Create completes
- **THEN** `id` SHALL equal `<cluster_uuid>/music`
- **AND** `connector_id` in state SHALL equal `music`

#### Scenario: connector_id captured after auto-generation

- **GIVEN** the resource is created without `connector_id`
- **WHEN** Create completes
- **THEN** `connector_id` in state SHALL equal the id returned by the API
- **AND** `id` SHALL equal `<cluster_uuid>/<auto-generated-id>`

#### Scenario: Data source ID set

- **GIVEN** the data source reads connector `music`
- **WHEN** Read completes
- **THEN** `id` SHALL equal `<cluster_uuid>/music`

### Requirement: `connector_id` requires replacement (REQ-003)

Changing `connector_id` SHALL require resource replacement. Elasticsearch does not support renaming connectors.

#### Scenario: connector_id change triggers replace

- **GIVEN** `connector_id` changes in configuration
- **WHEN** Terraform plans
- **THEN** replacement SHALL be required

### Requirement: Resource schema â€” envelope attributes (REQ-004)

The resource SHALL expose the following attributes for the connector envelope (sent in the narrow `PUT /_connector/{id}` body on Create and via `_name` / `_index_name` / `_service_type` / `_native` partial-update endpoints on Update):

- `service_type` (Required, String): connector service type, e.g. `"postgresql"`, `"mysql"`, `"github"`. The provider SHALL NOT validate this against an enum (new service types are added over time).
- `name` (Optional, String): human-readable connector name.
- `description` (Optional, String): description.
- `index_name` (Optional + Computed, String): destination Elasticsearch index name.
- `is_native` (Optional + Computed, Bool): `true` for Elastic-managed connectors, `false` for self-managed (default `false` on Elasticsearch's side).
- `language` (Optional, String): analyzer language.
- `api_key_id` (Optional, String): id of the API key used by the connector service for authorization.
- `api_key_secret_id` (Optional, String): id of the connector secret holding the API key (Elastic-managed connectors only).

#### Scenario: Envelope attributes round-trip through state

- **GIVEN** a connector created with `service_type = "postgresql"`, `name = "Music"`, `description = "catalog"`, `index_name = "music"`, `is_native = false`, `language = "en"`
- **WHEN** the resource reads state after create
- **THEN** all values SHALL match the configuration
- **AND** a subsequent plan SHALL show no changes

#### Scenario: name + description change triggers _name partial update

- **GIVEN** an existing connector with `name = "old"` and `description = "old"`
- **WHEN** the user updates `name = "new"` or `description = "new"`
- **THEN** the provider SHALL call `PUT /_connector/{id}/_name` with the new values

#### Scenario: index_name change triggers _index_name partial update

- **GIVEN** an existing connector with `index_name = "old"`
- **WHEN** the user updates `index_name = "new"`
- **THEN** the provider SHALL call `PUT /_connector/{id}/_index_name`

#### Scenario: service_type change triggers _service_type partial update

- **GIVEN** an existing connector with `service_type = "mysql"`
- **WHEN** the user updates `service_type = "postgresql"`
- **THEN** the provider SHALL call `PUT /_connector/{id}/_service_type`

#### Scenario: is_native change triggers _native partial update

- **GIVEN** an existing connector with `is_native = false`
- **WHEN** the user updates `is_native = true`
- **THEN** the provider SHALL call `PUT /_connector/{id}/_native`

### Requirement: Resource schema â€” pipeline nested block (REQ-005)

The resource SHALL expose an `Optional + Computed` `pipeline` nested attribute with the following fields:

- `name` (Required, String): ingest pipeline name.
- `extract_binary_content` (Required, Bool).
- `reduce_whitespace` (Required, Bool).
- `run_ml_inference` (Required, Bool).

Changes to `pipeline` SHALL trigger `PUT /_connector/{id}/_pipeline` with the full pipeline object. Defaults populated by Elasticsearch SHALL appear in state after Read.

#### Scenario: Pipeline create

- **GIVEN** `pipeline = { name = "search-default-ingestion", extract_binary_content = true, reduce_whitespace = true, run_ml_inference = false }`
- **WHEN** Create runs
- **THEN** the provider SHALL call `PUT /_connector/{id}/_pipeline` with that body

#### Scenario: Pipeline omitted on create populates defaults

- **GIVEN** `pipeline` is omitted from configuration
- **WHEN** Create completes and Read runs
- **THEN** state SHALL reflect the default pipeline populated by Elasticsearch
- **AND** a subsequent plan SHALL show no changes

#### Scenario: Pipeline update

- **GIVEN** an existing connector with `pipeline.run_ml_inference = false`
- **WHEN** the user updates `pipeline.run_ml_inference = true`
- **THEN** the provider SHALL call `PUT /_connector/{id}/_pipeline` with the full updated object

### Requirement: Resource schema â€” scheduling nested block (REQ-006)

The resource SHALL expose an `Optional + Computed` `scheduling` nested attribute with three sub-blocks:

- `full` (Optional, Object).
- `incremental` (Optional, Object).
- `access_control` (Optional, Object).

Each sub-block SHALL contain:
- `enabled` (Required, Bool).
- `interval` (Required, String): cron expression accepted by the Elasticsearch scheduler.

Changes SHALL trigger `PUT /_connector/{id}/_scheduling` with the full scheduling object.

#### Scenario: Scheduling create with all three sub-blocks

- **GIVEN** `scheduling = { full = {...}, incremental = {...}, access_control = {...} }`
- **WHEN** Create runs
- **THEN** the provider SHALL call `PUT /_connector/{id}/_scheduling` with all three sub-objects

#### Scenario: Scheduling sub-block omitted

- **GIVEN** `scheduling = { full = { enabled = true, interval = "0 0 0 * * ?" } }` (no `incremental`, no `access_control`)
- **WHEN** Create runs
- **THEN** the request body SHALL NOT include `incremental` or `access_control` keys
- **AND** state SHALL reflect both omitted sub-blocks as null

### Requirement: Resource schema â€” features nested block (REQ-007)

The resource SHALL expose an `Optional + Computed` `features` nested attribute with the following sub-blocks:

- `document_level_security` (Optional, Object with `enabled` Bool).
- `incremental_sync` (Optional, Object with `enabled` Bool).
- `native_connector_api_keys` (Optional, Object with `enabled` Bool).
- `sync_rules` (Optional, Object) with:
  - `basic` (Optional, Object with `enabled` Bool).
  - `advanced` (Optional, Object with `enabled` Bool).

Changes SHALL trigger `PUT /_connector/{id}/_features`.

#### Scenario: Features create

- **GIVEN** `features = { document_level_security = { enabled = true }, incremental_sync = { enabled = true } }`
- **WHEN** Create runs
- **THEN** the provider SHALL call `PUT /_connector/{id}/_features`

#### Scenario: Features update

- **GIVEN** an existing connector with `features.document_level_security.enabled = false`
- **WHEN** the user updates `features.document_level_security.enabled = true`
- **THEN** the provider SHALL call `PUT /_connector/{id}/_features`

### Requirement: Resource schema â€” `configuration_values` typed nested map (REQ-008)

The resource SHALL expose `configuration_values` as `Optional` `map(object({...}))` whose nested object has these fields:

- `string` (Optional, String).
- `number` (Optional, Number) â€” uses `NumberAttribute`; covers both int and float.
- `bool` (Optional, Bool).
- `json` (Optional, String with `jsontypes.NormalizedType{}`) â€” must be syntactically valid JSON.
- `secret_value` (Optional, String, **WriteOnly**, **Sensitive**) â€” write-only secret value.

A per-element `ObjectValidator` SHALL enforce that **exactly one** of `string`, `number`, `bool`, `json`, `secret_value` is set per element.

On Create / Update, the provider SHALL call `PUT /_connector/{id}/_configuration` with body `{"values": {<key>: <wire_value>}}` where each element compiles to a JSON value per the chosen branch:

- `string` â†’ JSON string.
- `number` â†’ JSON number.
- `bool` â†’ JSON boolean.
- `json` â†’ raw JSON pass-through.
- `secret_value` â†’ JSON string.

The body SHALL include every element present in the plan (including secret elements whose write-only value is read from `req.Config`, not state).

On Read, for each `configuration_values` element present in the prior state:
- If the API response's `configuration.<key>.sensitive == true`: the element SHALL be preserved from prior state (no refresh from API), and a warning diagnostic SHALL be emitted if the element's branch is NOT `secret_value`, instructing the user to move it.
- If the element's prior-state branch is `secret_value`: the `secret_value` field SHALL be preserved from prior state (write-only â€” has no refreshable state); other fields of that element SHALL remain absent.
- Otherwise: the API value SHALL be decoded into the same branch as the prior state (`string` stays `string`, `number` stays `number`, etc.).

For keys present in the API response but not in prior state (e.g. on import), the provider SHALL pick the branch matching the registered schema's `type` field: `"str"` â†’ `string`, `"int"` â†’ `number`, `"bool"` â†’ `bool`, `"list"` â†’ `string`. Sensitive fields SHALL be skipped on import (no value pulled into state).

The provider SHALL NOT remove API-side values for keys that the user has removed from `configuration_values`. The resource Read SHALL drop those keys from state silently. Documentation SHALL note that removing a key stops managing it but does not unset it server-side.

#### Scenario: Each branch round-trips through state

- **GIVEN** `configuration_values = { a = { string = "x" }, b = { number = 5432 }, c = { bool = true }, d = { json = jsonencode({ k = "v" }) } }`
- **WHEN** Create runs and Read populates state
- **THEN** each element's chosen branch SHALL match the configuration
- **AND** a subsequent plan SHALL show no changes

#### Scenario: secret_value is marked write-only and sensitive

- **GIVEN** `configuration_values = { password = { secret_value = "pw" } }`
- **WHEN** the resource is planned
- **THEN** plan output SHALL NOT include the value
- **AND** state SHALL NOT contain the `secret_value` field for this element

#### Scenario: Validator rejects multiple branches set per element

- **GIVEN** `configuration_values = { x = { string = "x", number = 1 } }`
- **WHEN** Terraform validates
- **THEN** validation SHALL fail with a diagnostic indicating exactly one branch must be set

#### Scenario: Validator rejects no branches set per element

- **GIVEN** `configuration_values = { x = {} }`
- **WHEN** Terraform validates
- **THEN** validation SHALL fail with a diagnostic indicating exactly one branch must be set

#### Scenario: Configuration write uses single _configuration call

- **GIVEN** `configuration_values` has three elements
- **WHEN** Create or Update runs
- **THEN** the provider SHALL call `PUT /_connector/{id}/_configuration` exactly once with a `values` map containing all three elements

#### Scenario: Sensitive API field warns when set in a non-secret branch

- **GIVEN** an existing connector where the registered schema marks `password.sensitive = true`
- **AND** prior state has `configuration_values.password = { string = "pw" }`
- **WHEN** Read runs
- **THEN** the element SHALL be preserved from prior state without refresh
- **AND** a warning diagnostic SHALL be emitted naming `configuration_values["password"]` and instructing the user to move it to `secret_value`

#### Scenario: Removing an element stops management but does not unset server-side

- **GIVEN** prior state contains `configuration_values.tables = { string = "album,artist" }`
- **WHEN** the user removes the `tables` key from `configuration_values`
- **AND** Update runs
- **THEN** the `_configuration` PUT body SHALL NOT include `tables`
- **AND** state SHALL drop the `tables` key

### Requirement: Configuration-schema-not-yet-registered diagnostic (REQ-009)

If the user sets `configuration_values` but `GET /_connector/{id}.configuration` returns an empty object, the provider SHALL NOT call `PUT /_connector/{id}/_configuration` and SHALL return a structured diagnostic with the title "Connector configuration schema not yet registered" and a body explaining that the connector service must boot and register a per-service-type schema first, with a link to the API tutorial.

#### Scenario: Pre-flight check catches empty schema

- **GIVEN** a connector exists with `service_type = "postgresql"` and no connector service has yet booted
- **AND** `GET /_connector/{id}.configuration` returns `{}`
- **WHEN** the user applies `configuration_values = { host = { string = "x" } }`
- **THEN** the provider SHALL NOT call `PUT /_connector/{id}/_configuration`
- **AND** diagnostics SHALL include an error with title "Connector configuration schema not yet registered"

### Requirement: Resource omits runtime telemetry; data source exposes everything (REQ-010)

The resource schema SHALL NOT include: `status`, `last_seen`, `last_synced`, `last_sync_status`, `last_indexed_document_count`, `last_deleted_document_count`, `last_sync_scheduled_at`, `last_sync_error`, `last_access_control_sync_status`, `last_access_control_sync_error`, `last_access_control_sync_scheduled_at`, `last_incremental_sync_scheduled_at`, `error`, `filtering`, `custom_scheduling`, the full `configuration` schema document (only `configuration_values` is exposed for write), `sync_cursor`, or `sync_now`.

The data source SHALL expose ALL of the above as `Computed` attributes, plus everything the resource exposes (with all attributes `Computed`).

#### Scenario: Resource excludes runtime telemetry

- **GIVEN** the resource schema
- **WHEN** introspected
- **THEN** the listed runtime-telemetry attributes SHALL NOT be present

#### Scenario: Data source includes runtime telemetry

- **GIVEN** the data source reads an existing connector
- **WHEN** Read completes
- **THEN** state SHALL include `status`, `last_seen`, `last_synced`, `last_sync_status`, `filtering`, `custom_scheduling`, and the full registered `configuration` document

### Requirement: Write-only secret drift detection via private-state hash (REQ-011)

The resource SHALL detect drift on each `configuration_values["<key>"].secret_value` write-only attribute by storing a bcrypt hash of the most recently applied value in resource private state, using the shared `internal/utils/writeonlyhash` helper.

The provider SHALL:

1. In `ModifyPlan`: for each `configuration_values` element with `secret_value` set in config:
   - Compute the hash with `Hasher{resourceTypeName: "elasticsearch_connector"}.Compute(value)`.
   - Read the stored hash from private state under key `secret_hash:configuration_values["<key>"].secret_value`.
   - If no stored hash exists, treat as first apply â€” no drift signal.
   - If stored hash matches the computed hash: no drift.
   - If stored hash does NOT match: mark the resource as needing update and emit a warning diagnostic of the form: `"Detected a change to write-only attribute configuration_values[\"<key>\"].secret_value; the resource will be updated."` The diagnostic SHALL NOT include the value.
2. For each `configuration_values` element that is removed from the new configuration: clear the corresponding private-state hash entry.
3. After a successful Create or Update: store the bcrypt hash of every `secret_value` present in the applied configuration into private state under the corresponding key.
4. Read SHALL NOT modify private state.

#### Scenario: First apply baselines the hash

- **GIVEN** no prior private state for `configuration_values["password"].secret_value`
- **AND** the user applies `configuration_values = { password = { secret_value = "pw1" } }`
- **WHEN** Create completes
- **THEN** private state SHALL contain a bcrypt hash of `"pw1"` under key `secret_hash:configuration_values["password"].secret_value`

#### Scenario: Subsequent apply with same value produces no drift

- **GIVEN** the prior private-state hash matches `"pw1"`
- **AND** the user runs `terraform plan` with the same config
- **THEN** the plan SHALL NOT mark the resource as needing update on the basis of `secret_value`

#### Scenario: Subsequent apply with changed value detects drift

- **GIVEN** the prior private-state hash matches `"pw1"`
- **AND** the user runs `terraform plan` with `secret_value = "pw2"`
- **THEN** the resource SHALL be marked as needing update
- **AND** a warning diagnostic SHALL be emitted naming `configuration_values["password"].secret_value`
- **AND** the diagnostic body SHALL NOT contain the literal `"pw2"` or `"pw1"`

#### Scenario: Removing a secret element clears its private-state hash

- **GIVEN** the prior private-state hash exists for `configuration_values["password"].secret_value`
- **AND** the user removes the `password` key from `configuration_values`
- **WHEN** Update completes
- **THEN** the corresponding private-state hash entry SHALL be cleared

#### Scenario: Read does not modify private state

- **GIVEN** a private-state hash exists for `configuration_values["password"].secret_value`
- **WHEN** Read runs (without a Create/Update)
- **THEN** the private-state hash SHALL be unchanged

#### Scenario: Post-import baseline behaviour

- **GIVEN** a connector imported via `terraform import` (no private state present)
- **WHEN** the first refresh runs
- **THEN** no drift signal SHALL be emitted on `secret_value`
- **AND** the first subsequent apply SHALL baseline the hash

### Requirement: Import support (REQ-012)

The resource SHALL support `terraform import` using either the bare connector ID or the composite `<cluster_uuid>/<connector_id>` form. The import SHALL resolve the cluster UUID via the configured Elasticsearch connection.

#### Scenario: Import with bare connector ID

- **WHEN** the user runs `terraform import elasticstack_elasticsearch_connector.x music`
- **THEN** the resource SHALL be imported with `connector_id = "music"`
- **AND** `id` SHALL be set to `<cluster_uuid>/music`

### Requirement: Minimum Elasticsearch version (REQ-013)

The resource and data source SHALL gate against Elasticsearch 8.12.0 as the minimum supported version via `entitycore.VersionRequirement`. Older versions SHALL produce a clear diagnostic.

#### Scenario: Older Elasticsearch returns version diagnostic

- **GIVEN** the configured Elasticsearch is older than 8.12.0
- **WHEN** Terraform attempts to plan or apply
- **THEN** diagnostics SHALL include an error stating the resource requires Elasticsearch 8.12.0 or later

### Requirement: Required cluster privilege (REQ-014)

Documentation SHALL state that the resource requires the `manage_connector` cluster privilege for write operations and `monitor_connector` (or `manage_connector`) for read operations. API errors stemming from missing privileges SHALL be surfaced verbatim in diagnostics.

#### Scenario: Privilege-denied error is surfaced

- **GIVEN** the configured user lacks `manage_connector`
- **WHEN** the resource attempts Create or Update
- **THEN** diagnostics SHALL include the Elasticsearch error message indicating the missing privilege
