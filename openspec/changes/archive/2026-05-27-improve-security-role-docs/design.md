## Context

The provider's security role resources (`elasticstack_elasticsearch_security_role`, `elasticstack_kibana_security_role`) are functional but their documentation fails users trying to configure non-trivial ACLs. The existing examples use placeholder values (`"all"`, `["test"]`) that don't model least-privilege patterns. The valid Kibana feature privilege strings (e.g. `"discover"`, `"siem"`, sub-privileges like `"minimal_read"`) are entirely absent from the docs, forcing users to call `GET /api/features` themselves and parse the response to discover valid values. Issue #2000 captures this directly.

Two separate improvements are needed: better content (examples + privilege reference), and a mechanism to keep that content current as the Kibana feature set evolves across releases.

## Goals / Non-Goals

**Goals:**
- Replace generic placeholder examples with scenario-based `.tf` files that model real-world least-privilege patterns
- Add a standalone provider guide that explains the privilege model, embeds scenario examples, and includes a Kibana feature privilege reference table
- Add enforcement: a scheduled workflow that detects drift between the documented feature table and the live Kibana API and opens a self-healing PR

**Non-Goals:**
- No changes to resource schema, CRUD behaviour, or Terraform state
- No changes to acceptance tests
- The guide does not attempt to document every Kibana feature exhaustively — it covers a curated, commonly-used subset

## Decisions

### Guide as a separate page, not expanded resource docs

The resource doc pages are generated by `tfplugindocs` from `templates/resources/*.md.tmpl`. These pages appear inline on the Terraform Registry and are the most-visited entry point. However, they are poorly suited for narrative explanations: they're expected to be concise and schema-focused. Embedding 40-feature reference tables and multi-resource composition examples would make them unwieldy.

A separate guide page (`templates/guides/security-roles.md.tmpl`) allows narrative structure while the resource pages stay focused, linking to the guide with a short "See also" note. This pattern is already established in the repo (`elasticstack-and-cloud.md.tmpl`, `elasticstack-kibana-rule.md.tmpl`).

### Scenario example files replace, not supplement, existing generic examples

The current `resource-with-base.tf` and `resource-with-feature.tf` contributed to the original confusion. Leaving them alongside new scenario files would mean the weakest examples remain (tfplugindocs embeds all files alphabetically, and `resource-with-base.tf` sorts before `resource-data-analyst.tf`). They are removed and replaced.

The scenario files are the source of truth for the guide's embedded code blocks — the guide template uses `{{ tffile "examples/..." }}` directives to avoid duplication. This means example file renames break the guide template, which fails CI via `make check-docs`. Acceptable: file renames in this directory would be caught immediately.

### Machine-readable JSON file as curated features source of truth

The guide's Kibana feature privilege table needs to stay current as Kibana adds features across minor releases. Options considered:

- **Agent memory / workflow judgment**: rejected — without a persistent list, the workflow re-surfaces the same undocumented features every run. The agent would need memory state to suppress them, which is just the JSON file with extra indirection.
- **Fully exhaustive table (all features)**: rejected — there are ~40+ Kibana features including internal/plugin-specific ones. An exhaustive table would need constant updates and would confuse users with irrelevant entries.
- **Curated JSON file** (`scripts/security-role-docs/kibana-features.json`): chosen. A human-maintained list of `documented` feature IDs (which appear in the guide table) and `skip` feature IDs (features the team has decided not to document). Net-new features from the API that appear in neither list trigger the PR. Humans correct curation during PR review.

### PR-opening (self-healing) over issue-opening

The `kibana-spec-impact` workflow opens issues because detected kbapi changes require human judgment about schema/implementation follow-up. Drift in the feature privilege table does not: the fix is mechanical (add rows to the table, update `kibana-features.json`). An agent can produce a correct PR directly. Issue-opening would create toil with no benefit.

### Drift detection trigger: weekly + `generated/kbapi/**` push

Mirrors the `kibana-spec-impact` workflow pattern exactly. The kbapi is regenerated when STACK_VERSION changes, so changes to `generated/kbapi/**` on main indicate a new Kibana version has been ingested. Weekly schedule catches any gaps. No additional trigger infrastructure needed.

### Live stack in workflow via shared setup, not snapshot comparison

The pre-activation step needs to call `GET /api/features` to get the authoritative feature list. Static analysis of the kbapi is insufficient here — Kibana features are runtime-registered by plugins and not fully represented in the generated client types. The repo already has shared workflow infrastructure (imported via `shared/setup-dev.md`) that provisions a live Elastic stack. Using that directly is simpler than maintaining a separate features snapshot updated by the buildkite pipeline.

### Guide template generates feature table from JSON, not hardcoded markdown

The `kibana-features.json` file is the source of truth for both the drift detection workflow and the guide template. The template reads the JSON at docs-generation time to produce the feature table, ensuring the committed docs always match the JSON. This is implemented as a custom `tfplugindocs` template function or a pre-processing step in `make docs-generate`.

If the template tooling doesn't support JSON-driven table generation natively, the table is hand-maintained in the template but the JSON remains the drift-detection input. The pre-activation Go script validates consistency between the two.

## Risks / Trade-offs

- **Feature table maintenance burden**: The guide covers a curated subset (~15–20 features). New features appear a few times per major release cycle. The workflow handles detection; human review handles curation. Low ongoing burden.
- **PR churn from internal/ephemeral features**: The `GET /api/features` response may include plugin-specific or internal features. The `skip` list in `kibana-features.json` is the mechanism to silence these. Initial population of `skip` during implementation will need to review the full feature list at STACK_VERSION.
- **Guide discoverability**: Users hitting resource pages directly (the common path via Terraform Registry search) may not navigate to the guide. Mitigated by a prominent "See also: Security Roles Guide" link in both resource page templates.
- **Example file rename breakage**: Guide template `{{ tffile }}` directives create a coupling between the guide and the example file paths. Renames fail `make check-docs` in CI — acceptable as the catch mechanism.
