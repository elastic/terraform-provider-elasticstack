## Context

Users who apply a mix of explicit (Terraform-managed) and dynamic (auto-generated by Elasticsearch) field mappings on the same index experience spurious plan drift when running `terraform plan` across workspaces. Each workspace's state may contain different sets of dynamic mappings. Even though the existing `elasticstack_elasticsearch_index` resource has semantic-equality logic (`MappingsSemanticallyEqual`, `mergeMappingsForPlan`), the merge-state-into-plan approach accumulates dynamic fields in state over time and diverges across workspaces.

The user's workaround was to use the [restapi Provider](https://registry.terraform.io/providers/Mastercard/restapi/latest/docs) to issue `PUT /{index}/_mapping` directly — a clear signal that the desired model is managing mappings on an existing index without owning its lifecycle.

## Goals / Non-Goals

**Goals:**
- Provide a drift-free way to manage explicit Elasticsearch index field mappings independently of index lifecycle.
- Work on any index regardless of how it was created (ILM, data stream rollover, templates, `elasticstack_elasticsearch_index`, external tooling).
- Follow the established sub-resource pattern in this codebase.
- Reuse existing client functions and semantic equality helpers.

**Non-Goals:**
- Changing the existing `elasticstack_elasticsearch_index` resource semantics.
- Managed field removal (requires reindex; ES does not support it via `_mapping`).
- Data stream support (`write_index_only` parameter) — deferred to a future change.
- A companion data source for reading back index mappings.
- Conflict detection between `elasticstack_elasticsearch_index` and this resource.

## Decisions

### Decision 1: New standalone resource rather than modifying the existing index resource

**Rationale:** The user's core use case is managing explicit field mappings on indices that are not owned by Terraform. Adding an `explicit_mappings_only` flag to `elasticstack_elasticsearch_index` would not help this case because that resource still owns the index lifecycle (destroy deletes the index). A separate resource with no lifecycle dependency is the correct abstraction.

**Alternative considered (Approach B):** Add `explicit_mappings_only = true` to `elasticstack_elasticsearch_index`. Rejected because it does not address indices created outside Terraform and couples mapping management to index lifecycle management. See research comment for full evaluation.

### Decision 2: `mappings` accepts all top-level keys, not just `properties`

**Rationale:** Decided by the issue author. Users may need to manage `dynamic`, `_source`, `dynamic_templates`, `runtime`, etc. alongside field `properties`. Supporting all top-level keys requires no additional complexity (the `MappingsType` custom string type already handles arbitrary JSON) and avoids a future breaking change.

### Decision 3: Destroy is a no-op (remove from state only)

**Rationale:** Decided by the issue author. Elasticsearch does not support removing field mappings without a full reindex. Making destroy a no-op (remove from state only) is the correct behavior; the resource documentation must clearly state this. This pattern is consistent with append-only resources in other providers.

### Decision 4: Error on create if the target index does not exist

**Rationale:** Decided by the issue author. Consistent with the behavior of `elasticstack_elasticsearch_index_alias`. Users who need the index to exist before managing mappings should use `depends_on` or a resource reference to `elasticstack_elasticsearch_index`.

### Decision 5: Reuse `MappingsType` and `MappingsSemanticallyEqual` from `internal/elasticsearch/index/mappings_value.go`

**Rationale:** The existing custom string type already handles JSON normalization and semantic equality for mappings JSON. Using it for the new resource avoids duplicating normalization logic and ensures consistent drift detection behavior.

### Decision 6: ID format `{cluster_uuid}/{index_name}`

**Rationale:** Consistent with other sub-resource ID formats in this provider (`elasticstack_elasticsearch_index_alias`, `elasticstack_elasticsearch_data_stream_lifecycle`).

### Decision 7: Read intersects API response with user-declared keys

**Rationale:** The resource must track only what the user declared. On read, the provider calls `GET /{index}/_mapping`, then reconstructs the stored mappings JSON by keeping only the top-level keys present in the config. Dynamic extras (fields added automatically by Elasticsearch or by other tools) are silently dropped. This is the mechanism that eliminates drift.

**Implementation note:** The intersection must operate at the top-level key level (e.g. retain `properties` if the user declared `properties`, discard auto-generated fields within properties not declared by the user). The existing `MappingsSemanticallyEqual` helper handles semantic equivalence; the intersection logic must be implemented in the `read` step.

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| Destroy is a no-op — users may be surprised that `terraform destroy` does not revert mappings | Document clearly in resource description and registry page. Analogous to the `elasticstack_elasticsearch_index` behavior for `aliases` sub-blocks. |
| Silent conflict if both this resource and `elasticstack_elasticsearch_index` manage mappings on the same index | Left to user; ES itself merges mappings — no data loss. Document in description. |
| Read intersection at the top-level key level may miss nested-field granularity | Acceptable for this scope. Users who need field-level granularity can use `properties` with only the fields they care about. |

## Open questions

All open questions from the research comment have been resolved by the issue author (@tobio 2026-05-19):

- **Destroy semantics** → Remove from state only.
- **Data stream support** → Not yet (out of scope for this change).
- **Index existence at create time** → Error if the index does not exist.
- **Scope beyond `properties`** → All top-level keys.
- **Conflict detection** → Left to the user.

No blocking open questions remain.
