## ADDED Requirements

### Requirement: `xpack.uptime.alerts.monitorStatus` primary params struct (REQ-043)

The plan-time `params` validator for `rule_type_id = "xpack.uptime.alerts.monitorStatus"` SHALL use the **Uptime**-namespace generated struct (`kbapi.KibanaHTTPAPIsXpackUptimeAlertsMonitorstatusCreateRuleBodyAlerting`) as the container for `mustNewParamsSchemaSpecFromContainer`. Using the Synthetics-namespace struct (`kbapi.KibanaHTTPAPIsXpackSyntheticsAlertsMonitorstatusCreateRuleBodyAlerting`) is incorrect and SHALL NOT be used for this rule type.

The Uptime struct's `Params` field models:

- `availability` (optional object: `range`, `rangeUnit`, `threshold`)
- `filters` (optional union: object form with `monitor.type`, `observer.geo.name`, `tags`, `url.port` each as `*[]string`, or plain string form)
- `numTimes` (required float)
- `shouldCheckAvailability` (required bool)
- `shouldCheckStatus` (required bool)
- `search`, `stackVersion`, `timerangeCount`, `timerangeUnit`, `timerange`, `locations`, `isAutoGenerated`, `version` (all optional)

#### Scenario: Params with availability passes primary validation

- GIVEN `rule_type_id = "xpack.uptime.alerts.monitorStatus"` and `params` containing `availability`, `shouldCheckAvailability`, `shouldCheckStatus`, and `numTimes`
- WHEN plan-time validation runs
- THEN the provider SHALL NOT return a params validation error for the `availability` field

#### Scenario: Params with filters object passes primary validation

- GIVEN `rule_type_id = "xpack.uptime.alerts.monitorStatus"` and `params` containing `filters` as a JSON object with one or more of `tags`, `url.port`, `observer.geo.name`, `monitor.type`
- WHEN plan-time validation runs
- THEN the provider SHALL NOT return a params validation error for the `filters` field

#### Scenario: Synthetics-only fields are rejected

- GIVEN `rule_type_id = "xpack.uptime.alerts.monitorStatus"` and `params` containing a `condition` or `monitorIds` field (Synthetics-namespace only)
- WHEN plan-time validation runs against both specs in the registered list
- THEN the provider SHALL return a validation error (neither the Uptime struct nor the legacy fallback accept those fields)

### Requirement: `xpack.uptime.alerts.monitorStatus` legacy fallback `filters` completeness (REQ-044)

The `legacyMonitorStatusParams.Filters` struct used as the second (legacy fallback) spec for `"xpack.uptime.alerts.monitorStatus"` SHALL declare all four filter sub-fields that Kibana accepts, each as an optional pointer to a slice of strings with the corresponding dot-notation JSON key:

| Go field name   | JSON key            |
|-----------------|---------------------|
| `Tags`          | `"tags"`            |
| `MonitorType`   | `"monitor.type"`    |
| `ObserverGeoName` | `"observer.geo.name"` |
| `URLPort`       | `"url.port"`        |

All four fields SHALL be tagged `omitempty` so they do not appear as required keys in the `computeRequiredKeys` heuristic.

#### Scenario: Legacy params with multi-field filters passes validation

- GIVEN `rule_type_id = "xpack.uptime.alerts.monitorStatus"` and a `params` object that matches the legacy shape (`numTimes`, `shouldCheckStatus`, `shouldCheckAvailability` present) and `filters` containing one or more of `url.port`, `observer.geo.name`, `monitor.type`
- WHEN plan-time validation runs
- THEN the provider SHALL NOT return a params validation error caused by the extra filter sub-fields
