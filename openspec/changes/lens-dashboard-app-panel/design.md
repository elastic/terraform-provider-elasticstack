# Design: `lens-dashboard-app` Panel Support

## Context

The Kibana Dashboard API exposes a panel type `lens-dashboard-app` that is distinct from the existing `vis` panel type used by the provider's typed Lens converters. The generated `kbapi` model is:

- `KbnDashboardPanelTypeLensDashboardApp` with top-level `grid`, optional `id`, `type = "lens-dashboard-app"`, and required `config`.
- `config` is the union `KbnDashboardPanelTypeLensDashboardApp_Config`.
- By-value config is `KbnDashboardPanelTypeLensDashboardAppConfig0`, which is an opaque union over Lens chart schemas such as `metricNoESQL`, `metricESQL`, `xyChartNoESQL`, `xyChartESQL`, `pieNoESQL`, `treemapESQL`, and `waffleESQL`.
- By-reference config is `KbnDashboardPanelTypeLensDashboardAppConfig1`, an object with required `ref_id` and `time_range`, plus optional `references`, `description`, `title`, `hide_title`, `hide_border`, and `drilldowns`.

The previous proposal described older field names (`attributes`, `saved_object_id`, and `overrides`) that are not present in the current generated API model. This design aligns the Terraform shape with the current model.

## Goals

1. Allow practitioners to embed saved Lens visualizations by reference using the API's `ref_id` and `references[]` fields.
2. Allow practitioners to embed Lens visualizations by value using a JSON object that maps directly to the generated by-value `config` union.
3. Expose the by-reference API fields with idiomatic Terraform names where possible.
4. Enforce mutual exclusivity between `by_value` and `by_reference` sub-blocks at plan time.
5. Keep the implementation consistent with the existing dashboard panel architecture.

## Non-Goals

- Supporting `lens-dashboard-app` panels through the existing panel-level `config_json` attribute.
- Providing typed Terraform blocks for every by-value Lens chart schema in this change.
- Translating old `attributes` / `saved_object_id` / `overrides` shapes that are not in the current generated API model.
- Changing any existing `type = "vis"` Lens panel behavior or typed Lens config block.
- Migrating existing `vis` Lens panels to `lens-dashboard-app`.

## Decisions

### Terraform shape: `lens_dashboard_app_config` with `by_value` and `by_reference` sub-blocks

The panel config is modeled as a typed block `lens_dashboard_app_config` containing two mutually exclusive nested blocks: `by_value` and `by_reference`.

```hcl
lens_dashboard_app_config = {
  # Exactly one of by_value or by_reference must be set.

  by_value = {
    config_json = string # required; JSON object matching one generated by-value Lens chart config schema
  }

  by_reference = {
    ref_id          = string # required; API ref_id for the linked library item
    references_json = string # optional; JSON array of { id, name, type } reference objects
    title           = string
    description     = string
    hide_title      = bool
    hide_border     = bool
    drilldowns_json = string # optional; JSON array matching the generated drilldowns union

    time_range = {
      from = string # required
      to   = string # required
      mode = string # optional; absolute | relative
    }
  }
}
```

The block is optional and mutually exclusive with all other panel config blocks, enforced by REQ-006.

### By-value: `config_json` maps directly to API `config`

`by_value.config_json` accepts a JSON object that is sent directly as `KbnDashboardPanelTypeLensDashboardApp.config`. The JSON must match one of the generated by-value Lens chart schemas. This includes the chart's own `type`, required chart fields, required `time_range` where the selected schema requires it, optional `references`, and optional display fields such as `title`, `description`, `hide_title`, and `hide_border`.

The provider should not wrap this JSON in an `attributes` object and should not split `references` out into a separate Terraform attribute for by-value mode. The generated API model places those fields inside the by-value chart config object.

### By-reference: `ref_id`, `references_json`, and required `time_range`

The current by-reference API object requires `ref_id` and `time_range`. It does not expose `saved_object_id` or `overrides`.

`ref_id` is the API reference name used by the dashboard panel. `references_json` carries the API `references` array using `KbnContentManagementUtilsReferenceSchema` objects with `id`, `name`, and `type`. For a saved Lens visualization, practitioners should set a reference whose `name` matches `ref_id`, whose `type` is the saved object type (typically `lens`), and whose `id` is the saved object ID.

`time_range` is required in the Terraform by-reference block because it is required by `KbnDashboardPanelTypeLensDashboardAppConfig1`. The nested block exposes `from`, `to`, and optional `mode`, matching `KbnEsQueryServerTimeRangeSchema`.

### By-reference optional fields

`title`, `description`, `hide_title`, and `hide_border` map directly to the by-reference API object. `drilldowns_json` is an optional JSON array escape hatch for the generated drilldowns union. It is kept opaque because the generated model contains multiple drilldown variants and introducing typed Terraform blocks for them is outside this change.

### Panel-level `config_json` support for `lens-dashboard-app`

The existing panel-level `config_json` write path is not extended to `lens-dashboard-app`. Reasons:

1. By-reference requires typed fields (`ref_id`, `time_range`) and reference handling that should not be hidden behind the generic panel fallback.
2. By-value already has a raw JSON escape hatch inside `lens_dashboard_app_config.by_value.config_json`.
3. Keeping `lens-dashboard-app` behind a dedicated typed block avoids ambiguity with the existing `config_json` support for other panel types.

**OpenSpec traceability:** REQ-010 is updated for panel-level `config_json` allowlist and typed-only round-trip wording; REQ-025 is updated for the raw `config_json` authoring and write diagnostic. Both align with the proposal and delta (use `lens_dashboard_app_config` for `lens-dashboard-app`).

### Relationship to existing `vis` Lens panel type

The existing typed Lens config blocks (`xy_chart_config`, `metric_chart_config`, `waffle_config`, etc.) continue to apply exclusively to `type = "vis"` panels. They are not valid for `type = "lens-dashboard-app"`. This distinction is enforced by the type-specific validation rules in REQ-006.

Practitioners with existing `vis` Lens panels do not need to migrate. The two panel types coexist. The `lens-dashboard-app` type is the correct choice when referencing a saved Lens visualization through the dashboard application panel model or when importing/exporting dashboards that use that panel discriminator.

### Read-back and drift prevention

On read-back, the provider classifies the API `config` **JSON object from its raw shape** (REQ-035). Decode-as-`Config1` alone is not the rule: the generated union helpers are plain `json.Unmarshal` and do not enforce a true oneOf on the wire. The order is:

1. **By-value (precedence):** If the object has a non-empty string at top-level `type` (the by-value Lens chart discriminator), preserve the full `config` under `by_value.config_json`, including when `ref_id` and `time_range` are also present on the same object.
2. **By-reference:** If that chart discriminator is **absent** and the object has non-empty `ref_id` and a `time_range` with non-empty `from` and `to`, populate `by_reference`.
3. **Ambiguous / incomplete:** If the response matches neither pattern and prior plan or state had `by_reference`, preserve that prior `by_reference` block per REQ-009 rather than silently mode-flipping to `by_value`.
4. **Fallback:** Otherwise preserve the full `config` under `by_value.config_json`.

`by_value.config_json`, `by_reference.references_json`, and `by_reference.drilldowns_json` should use semantic JSON equality so API ordering changes do not create spurious plan diffs. For by-reference optional scalar fields, read-back should seed from prior practitioner intent where the API omits optional values, consistent with existing panel drift-prevention behavior.

**By-value practitioner preservation:** When Kibana’s stored `config` is a **value-superset** of the practitioner’s planned `by_value.config_json` (same values on every path the practitioner set, with extra keys/defaults on read), the implementation preserves the original Terraform string in state so apply/refresh stays stable (see REQ-035 and `preservePriorLensByValueConfigJSON` in `internal/kibana/dashboard/models_lens_dashboard_app_converters.go`).

**Arrays (intentional limit):** JSON arrays are matched by index. The API may have **trailing** elements the practitioner did not author (length `len(current) > len(prior)` with a matching prefix) or, for an empty practitioner array, the API may fill in new items, which **fails** subset match so state follows the read-back and avoids wiping API-only metrics. The logic does **not** treat **reordered** or **prepended** array entries as the same as the practitioner’s array—only prefix alignment at indices `0..len(prior)-1`.

## Risks and Trade-offs

| Risk | Mitigation |
|------|------------|
| `by_value.config_json` is opaque JSON; practitioners must know the generated Lens chart schema | Document that it maps directly to the `lens-dashboard-app.config` by-value union and can include the chart `type`, `references`, `time_range`, and display fields required by that schema |
| `ref_id` / `references_json` is less direct than `saved_object_id` | Align with the generated API and document the common reference pattern (`name = ref_id`, `type = "lens"`, `id = saved object ID`) |
| By-reference `time_range` is required by the current API even when practitioners think of it as an override | Make the Terraform block required in by-reference mode to catch invalid payloads before the API call |
| `drilldowns_json` is opaque | Keep it as a JSON escape hatch until there is a separate reason to model dashboard drilldown variants |

## Migration and State

This change is purely additive. No existing dashboard state is affected. There is no schema version change required for the dashboard resource. Practitioners who attempted to manage `lens-dashboard-app` through panel-level `config_json` will need to migrate to `lens_dashboard_app_config`.

## Open Questions

1. **Reference type value**: Confirm whether saved Lens visualization references must always use `type = "lens"` in the dashboard API, or whether Kibana can emit another saved object type for some Lens library items.

2. **Read-back classification (resolved in implementation; residual API caveat):** REQ-035 and the provider use the raw-shape classifier above (root chart `type` precedence, then by-reference when there is no chart discriminator and `ref_id` plus `time_range.from`/`to` are present, then ambiguous handling with prior `by_reference` per REQ-009, then fallback to `by_value`). **Residual:** If a future or edge-case Kibana payload is a true by-value chart but omits a top-level `type` string, or otherwise falls outside these patterns, classification may not match practitioner intent until the config is corrected in Kibana or the rules are extended; the ambiguous path limits silent mode flips for managed `by_reference` panels.

3. **By-reference optional defaults**: Confirm whether Kibana injects `hide_title`, `hide_border`, or default drilldown fields on read-back so the implementation can preserve practitioner intent where needed.
